# 設計の背骨 — 「選択と能動で一周の形が変わる」(R34・2026-07-21 ユーザー確定)

ユーザー確定の芯: **選択(どの型で回すか・道中どう動くか)と能動(タップ・討伐・金・タイミング)で、一周の"形"が変わる。**
均一(全稼ぎ口が薄く等分)を良しとせず、型ごとに山と谷をわざと作る。

## 0. 現状診断(なぜ空っぽか)
30個の合格条件は「どれも死んでない」ことは保証するが「どれかが生きてる」ことは一個も保証しない。
特に **㉘(各稼ぎ口≥10%を全周回)** と **②改(方針間liftを±1.5帯に収める)** が、全型・全周回を同じ指数カーブへ均す。
結果、hunt周回とbake周回が「数字の帰属先が違うだけ」で体験が同じ＝選んでる感が無い。能動的な瞬間(タップ/討伐/金/ボス)は
"起きて終わる"だけで、プレイヤーの動きで結果が変わらない＝autobuy仕込んで眺めるだけになる。

## 1. 二本柱

### 柱A: 選択が一周の形を変える(型 = 支配的な稼ぎ口 + 固有の遊び)
- 型を選ぶと、その型の**署名稼ぎ口が支配的**になってよい(均さない)。他は「死んでなければいい」だけ。
- 各型に**その型でしか回らない固有の能動ループ**を1本持たせる。
  - **hunt(狩猟)**: 討伐そのもの。最小タップ床で「戦う」実体が戻った。狙う相手を型(affinity)で選ぶ→素材/報酬が変わる。
    残HP×5のオーバーキルでレア。積極的に狩る周回ほど素材・討伐直送が伸びる。形＝戦闘で山ができる。
  - **golden(金色)**: 金クッキー連鎖。短時間に続けて取ると連鎖倍率が乗る。能動的に窓を作る。形＝バースト。
  - **click(会心)**: タップ+会心。会心は「序盤低く・たまにデカく・僅かに増える」(ユーザー確定形)。連打すると稀に大会心。形＝手が忙しい山。
  - **bake(焼成)**: 放置生産のランプが主役。能動は「どの生産ラインに寄せるか」の一手。形＝なだらかな指数(=これはこれで一つの快感)。
  - **balanced(標準)**: 四稼ぎ口を回す。編成の心得(ensemble)=全部を生かすほどボーナス。形＝広く薄く、ただし全部触る忙しさ。

### 柱B: 能動 > 放置(積極的に遊ぶと、その型の得意軸で伸びる)
- 各型で「その署名ループを能動的に回す」と、autobuy放置よりはっきり伸びる差を作る。
- 差は署名軸に集中させる(hunt=討伐由来、golden=金、click=タップ/会心、bake=生産)。放置でも回るがランプは低い天井。

## 2. 合格条件への影響(=ユーザーへの提案が要るもの)
柱Aは **㉘の定義**(全稼ぎ口≥10%)と正面衝突する。均一を強制する限り「型が形を変える」は原理的に不可能。
- **提案㉘'(要承認・定義変更)**: 「各型の署名稼ぎ口 ≥ 支配閾値(例35%)、かつどの稼ぎ口も ≥ 死なない床(例5%)」へ。
  = hunt周回は討伐が支配的でいい・golden周回は金が支配的でいい。等分の強制をやめ、「支配 + 死なない」に。
- **提案②改'(要承認)**: 方針間liftの±帯は「型が違えば得意軸のliftが大きく違ってよい」向きへ緩める。均一化圧を外す。
- **提案㉛(新設・要承認)**: 「能動>放置」——各型で署名ループを回すプレイが、同型の放置プレイより署名軸で ≥ K倍(例1.5)。

**重要**: 条件は俺の判断で勝手に変えない(ユーザーの所有物)。上記は提案として明示し、
実装時は**旧条件(㉘等)も並走して測り続ける**(壊れたら見える)。新レンズ(㉘'/㉛)でチューニングし、
戻ってきたユーザーに旧⇔新の対比データを出して承認を仰ぐ。承認前に旧条件を破壊しない。

## 3. 作る順序(measure → design → build → verify、各段でbranchにコミット)
1. **measure**: runnerに新モード`shape`——型ごとに(署名稼ぎ口シェア / 能動vs放置の署名軸乖離 / 形の山谷)を出す。
   これで「今どれだけ均されてるか」を数値で見る。ここが今の起点。
2. **design判断**: 各型の署名ループの効き所を、既存レバー(直送・otherMul・killValMul・golden連鎖・crit・own指数)で
   どこまで「支配 + 能動優位」に寄せられるか。新機構は最小限、まず既存の再配分で形を作る。
3. **build**: 型ごとに署名軸を太らせ・他を床まで許容。能動ループの報酬を署名軸へ寄せる。
4. **verify**: `shape`(新)+ baseline/checks/income/expect(旧)を毎回。旧が崩れたら「提案で緩める」候補として記録、
   勝手には均し直さない。装備(a)(b)・⑬・㉑等の非署名条件は維持。

## 3.5 測定で判明した現状(2026-07-21 shape/income/checks・最小タップ床込み)
- **署名支配度**: golden 61% / bake 62% / hunt 45% は既に署名が支配的=形あり。
  **click タップ20%=最弱**(署名が支配できていない)。**balanced 最小稼ぎ口4%**(breadth=全≥10%が崩れ)。
- **⑫ 方針間の総合効率 最良/最悪=96×(≤3が基準)**: 型ごとの到達総量が桁違い(golden系S3=e84・状況S11=e91 vs
  bake系S1=e281)。= **「どの型を選んでも成立する」が壊れている=選択が本物になっていない**。⑫は新方向と一致=残す。
  ※これは最小タップ床由来ではなく構造的(golden/状況型は転生回数が少なく累積が伸びない)。
- **⑧未達→転生 全方針OK**=最小タップ床は転生ループ無害。**⑬熟成1164×NG**は戦闘無関係=別件として切り分け(要調査)。
- **優先順位(build)**:
  1. **方針到達量のパリティ(⑫≤3)**: golden/状況型を他型と comparable な総量へ。選択を本物にする最重要。
  2. **click の署名支配**: タップ20%→~35%。ただし tapDirect の係数墓場(clickBonusLate/satMaxLate/anchorGolden/
     otherMulLate…=均す機械)に更に係数を積むのは轍。**能動的な瞬間=会心(稀で大)**を click の支配源にする方向を優先検討。
  3. **balanced の breadth**: 最小稼ぎ口≥10%。
  4. **能動優位**(bnfzzi8l7の改良版データ待ち): 各型で積極プレイが署名軸で伸びるか。

## 4. やらないこと(momentum/最小タップ床の教訓)
- 一個の倍率で全成長を乗っ取る設計は二度としない(削除耐性ゼロ+単調化)。成長は複数の見える機構へ分散。
- 「面白いはず」で頼まれてない機構を勝手に足して既成事実化しない。署名ループの各要素は、この背骨(柱A/B)に
  照らして「なぜそれが形を作るか」を言えるものだけ入れる。言えないものは入れない。
