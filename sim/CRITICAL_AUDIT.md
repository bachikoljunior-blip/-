# 批判監査(あらゆることに問題をあぶり出す仕組み・2026-07-24 ユーザー指示)

指示: 「常にあらゆることに対して批判して問題をあぶり出せ。改善ループの仕組みをあらゆることに適用しろ」。
=毎ループ、まず**「これは欠陥だ」と仮定して反証を試みる**(健全確認から入らない)。対象はゲームだけでなく**自分の成果物・手法・主張・仕組みそのもの**を含む。

## 姿勢(反証ドリブン)
- 「0 errors / 健全」で終わらせない。0が出たら**測り方が浅い**を疑う(例: pageerrorしか見ずconsole.error/warnを見逃していた=下記F1)。
- 自分の直近成果を最初に叩く: 「この主張は過大では?」「この忠実性は本物か?」「このBOTは実プレイヤか?」。
- 設計判断(経済/ペーシング)は勝手に直さず、**証拠付きで批判をユーザーへ上げる**(sim先行・平均保存の制約)。純UI/文言/ツール/手法の欠陥は自己判断で即修正。
- 過大主張しない: 「ゲームが悪い」と「自分のBOT/測定が弱い」を切り分けてから断じる。

## 監査の切り口(毎回この観点で叩く)
1. **手法の穴**: 検証が浅い所(監視漏れ・偽陰性・カバレッジ抜け)。
2. **成果物の過大主張**: 実況/レポートが実体より良く見せていないか(最適プレイを"一般プレイヤー"と偽っていないか)。
3. **ゲームのペーシング/到達性**: コンテンツが実プレイヤに届くか(死にコンテンツ)。
4. **UX/文言の不明瞭**: プレイヤが意味を取れない表示。
5. **felt/演出**: 単調・見分けつかない・気持ちよくない箇所。
6. **頑健性/依存**: 外部依存・エラー処理・エッジ破綻。

## 発見台帳(重大度順・状態を更新)
- **F1 [手法・確定]** QAが `pageerror` しか監視せず `console.error`/`console.warning`/`requestfailed` を見逃していた=「0エラー」報告が不完全だった。実測: 起動毎に Firebase CDN 読込失敗の console error が1件出ていたのに全QAで未検出。→**対策**: QAハーネスに console/requestfailed 監視を追加(下記F6の関連)。状態: stage_playthrough に追加中。
- **F2 [依存・要検討]** ゲームが `gstatic.com/firebasejs/...` を外部CDNから読む=ネット遮断/CDN障害でクラウドセーブ機能が落ちる(この環境では毎回失敗、ゲーム自体はオフライン継続=graceful degradeは確認)。批判: 外部依存の可視化/失敗時のユーザー通知は? 自己ホスト/内蔵の検討余地。要ユーザー判断。
- **F3 [ペーシング・要検討/要証拠]** 転生費は 1e9→1e17→1e25→1e31→1e40…と ×1e8/回、`prestigeMultiplier=1+skillEffect(all)` は有限スキルで頭打ち。実測でステージ4到達に非現実的な実プレイ時間(~1e26日)。**批判仮説**: ステージ3-6・装備・注文・料理=ゲームの大半のコンテンツが実プレイヤに事実上未到達=死にコンテンツでは? **未確定**: これが「素の長期設計」か「自分のBOTのcps複利が弱い(momentum/装備/最適購入を使っていない)」かを切り分けていない。→要: 強いプレイモデルでcps上限を測り、ゲーム設計 vs BOT弱さを分離してからユーザーへ上げる。
- **F4 [成果物の過大主張・確定]** stage_playthrough は「一般的な人間プレイヤー」を謳うが、実際は**最適プレイ**(金クッキーを毎回全回収・モンスター全狩り・値/費用の貪欲最適購入・放置も最適)。一般プレイヤーは取りこぼす。→実況が実体より活発/最適に見える。要: 「最適寄りの上限プレイ」と正直に明記 or カジュアル係数を入れる。
- **F5 [UX文言・確定]** 金クッキーの効果サフィックス `金 +X / 香料棚` は**源(香料棚)を名指すだけで効果(全生産バースト30秒)を言わない**=プレイヤが何を得たか分からない。要: 効果を明示("香料棚: 全生産バースト30秒"等)。純文言=自己修正可。
