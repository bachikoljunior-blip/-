## 1. プロジェクトの全体像

- このリポジトリは日本語のブラウザゲーム「クッキーストラテジャー」(クッキークリッカー系放置ゲーム)です。
  ゲーム本体はルートの `index.html` 1ファイル(HTML+CSS+JSすべて込み)。他の画像類は素材です。
- ゲームの構造: クッキーをタップ/設備(強い指→おばあちゃん→オーブン→…→反物質オーブンの15種)で生産し、
  「研究」(13種)で強化、「ノルマ」(時間で増える必要量を維持するとモンスターが出る)、
  「金クッキー」「モンスター討伐報酬」、そして「転生」(周回リセットで転生PTを獲得し、スキルツリー68ノードを買う)がある。
- 依頼主の目的: **100時間×10種類のプレイ方針でシミュレーションし、決められた合格条件をすべて満たすようにゲームの数値・機能を調整する**こと。
  これまで第1〜6次の調整が行われた(mainの履歴参照)。現在の `index.html` は第6次(研究共鳴システム入り)。

- 代わりに「**段階式研究**」(後述の13研究×3段階)を実装する。これは以前提案して承認された内容。
- 転生には**所持クッキー100万が必要**(転生時に消費)。これは実装済み仕様で維持する。
- 作業ブランチは `claude/cookie-game-balance-k456aa`。ゲームコードの変更はユーザー承認後にPRを作る。

## 2. シミュレーション環境(このディレクトリ)

ゲームJSとは独立の Node.js 製シミュレータでバランスを検証する。ゲームの式を忠実に移植したもの。

| ファイル | 役割 |
|---|---|
| `sim.js` | ゲームエンジン移植(生産・ノルマ・金・モンスター・研究(段階式)・転生・スキル)。1秒刻み、乱数はすべて期待値化 |
| `params.js` | 調整パラメータ(この値を変えて調整する)。**現在はチューニング済みの状態** |
| `strategies.js` | 10種類のプレイ方針(S1〜S10)。**プレイヤー視点の情報のみで設計する(合格条件は一切考慮しない)。ゲームを調整したら、そのたびに「調整後のゲームをプレイヤーとして見たらどう遊ぶか」で作り直す(下記3-4)** |
| `runner.js` | 実行・判定ハーネス。`node runner.js baseline "" 100` / `detail S1 100` / `pacing S1 100` / `toggles S1 100` / `diag S1 100` / `profile S1 100 <run番号>` |
| `tune.js` | スキルコスト自動チューナ(`node tune.js 12 S1 [--resume]`)。周回時間を帯域に合わせて56ラングのコストを反復調整し、ベストを `rung_costs.json` に保存 |
| `rung_costs.json` | チューニング済みスキルコスト(56メインラング)。`params.js` の `skillCost.rungCosts` に焼き込み済み |
| `gen_html7.py` | HTMLパッチ生成スクリプト。**注意: 入力パスがアップロード一時ファイルを指しているため、そのままでは動かない。**リポジトリの `index.html` を対象に書き直すか、パッチ一覧の参考として使う |

- 実行方法: `cd sim && node runner.js baseline "" 100`(10方針×100時間、数分かかる)
- 検証の考え方(確立済み): 研究・報酬などの有効/無効比較は「**1つだけ効果ゼロ**(購入などの行動は同一)」にした100時間シムを回し、
  **同一スキル構成の周回同士**で「獲得効率(周回総クッキー÷周回時間)」の比を取り幾何平均する(`runner.js` の `toggles`)。

### 3-4. 作業ループとプレイ方針の決め方(ユーザー指定・最重要)

プレイヤーは調整後のゲームを見て遊び方を変える。そのため作業は次のループで回す:

```
(1) ゲームの数値・機能を調整する
(2) 調整後のゲームを「プレイヤーとして」見て、10種類のプレイ方針を考え直す(作り直す)
(3) その方針で100時間×10方針のシミュレーションを実行する
(4) 合格条件①〜⑬を判定する
(5) 不合格があれば (1) に戻る(そのたびに (2) の方針見直しも必ずやり直す)
```

**プレイ方針の決め方のルール(第1次からのユーザー指定)**:
- 方針は10種類。プレイヤーのタイプが分かれるように作る(バランス型/クリック特化/金クッキー特化/狩猟特化/研究貯蓄/早回し転生/長期育成/最新設備ラッシュ/ノルマ死守/のんびり放置、といった多様なアーキタイプ)。
- **プレイヤーから見える情報だけ**で設計する: 設備・研究のコストと所持数、現在と次の増分、ノルマ表示、報酬プレビュー、転生PTの見込み表示など。内部式やシミュレーション結果は見ない。
- **シミュレーションの合格条件は一切考慮しない**。方針の変更理由は常に「調整後のゲームをプレイヤーとして見たらこう遊ぶのが得だから」であること。合格条件の達成度から逆算して方針を作るのは禁止。
- 各方針の判断基準は数値で明記する(例: 「研究はコスト≤所持30%なら購入」「獲得見込みPT≥次スキル最安コスト×1.2 かつ 経過120秒以上で転生」)。
- 狙いは各プレイヤーなりに「全周回の総クッキー合計の最大化」。
- 転生には所持クッキー100万が必要というゲームルールには全方針が従う。
- 乱数要素はすべて期待値で扱う(金クッキーは即時獲得/ブーストを交互、報酬候補は決定的ローテーション、会心は期待値倍率)。
- 「回」= 転生から転生まで(第0回 = 開始→初転生。100時間で終わる途中の周回は「途中」扱い)。

## 4. これまでの調整で分かった重要な技術的知見(再発見に時間を溶かさないこと)

1. **固定倍率は効かない(垂直吸収)**: 周回終盤の成長は毎秒0.3桁(10進)を超える垂直成長になるため、
   生産に定数倍率を掛けても周回時間は数秒しか変わらず、有効/無効比は1.0に張り付く。
   機能を効かせるには「所持数や周回内変数が**成長率**に入る」形(指数の肩、発動系、成長ゲート)が必要。
2. **周回時間のモデル**: 周回時間 D ≈ A(深さ) + B(ラング)×(しきい値 − 自然フロンティア)。
   A=再登坂時間(過去に到達した深さまで戻る時間、深さにほぼ線形 A≈3.1×桁数)。
   B=未踏領域の開拓コスト(秒/桁)。**Bはラング13で約1,000〜1,400秒/桁、ラング25以降は10〜60秒/桁**まで落ちる。
   スキルしきい値を上げても、壁の手前では時間がほぼ動かず、壁を越えると突然何時間もかかる(ラング13でe125を要求したら25時間かかった実測あり)。
3. **×100(④)のノイズ**: 終盤の垂直成長では1秒の判定粒度+金ブースト重なりで周回総クッキーが±1〜1.7桁ぶれる。
   隣接周回比を安全に≥100にするには、しきい値間隔に2.6桁以上の余裕を取り、
   **設備コストの膝(knee=4300個、超過分指数1.0)で周回末端の速度を抑える**のが有効だった(×100が55/55で安定)。
4. **転生PT式は pG=0.80**(gain=0.4×(周回クッキー/1e4)^0.80)に変更済み。×100には コスト比R ≥ 100^0.80 ≈ 40 が必要。
5. **構造的に不可能な領域**(旧帯域式での実測。新式でも本質は同じなので説明に使う):
   - 第0回: 100万クッキー到達に最短20分超、帯域上限は数分 → 不可(ユーザー了承の「クリック」例外)。
   - 初回アセント(第1〜12回目安): 未踏コンテンツの開拓が B≈1,000〜4,800秒/桁で、×100に必要な2桁すら帯域内に収まらない。
   - 開拓直後の数周回: 高速再登坂領域で0.5Yより短くなりがち(押し込みは壁で頭打ち)。
   - クリック会心型(S2): クリックが線形スケールのため初回アセントが他方針の3〜4倍遅い(5-2参照)。

## 5. ユーザーに提案済み・承認待ちの2案(このセッションでは実行しない指示だった)

### 5-1. 帯域式(⑥⑦のY): √型を推奨として提示済み
- **推奨: Y(x) = 120 + 8×√x** (x=経過秒)。序盤2分から始まり、伸び率 dY/dx=4/√x が時間とともに減衰。
- 比較表(帯域は [0.5Y, Y]):

| 経過時間 | √型(推奨) | 飽和型 120+5400x/(x+90000) | 対数型 120+300ln(1+x/1200) |
|---|---|---|---|
| 開始直後 | 2分00秒 | 2分00秒 | 2分00秒 |
| 10分 | 5分16秒 | 2分36秒 | 4分02秒 |
| 1時間 | 10分00秒 | 5分28秒 | 8分56秒 |
| 6時間 | 21分36秒 | 19分25秒 | 16分43秒 |
| 24時間 | 41分12秒 | 46分05秒 | 23分27秒 |
| 100時間 | 1時間22分 | 1時間14分(上限92分に漸近) | 30分32秒 |

- 選定理由: 全スキル解放までの周回数が55〜70回程度に収まり×100×周回数の積が有限に収まる/序盤に十分な幅がある。
- **ユーザーがどれを選んだかを確認してから runner.js / tune.js の yCurve を書き換えること。**

#### 帯域の構造問題は式を変えても自動では直らない(重要)
ゲームの周回時間の自然な軌跡は「初回アセントで山(30分〜5時間)→壁の手前で谷(2〜6分)→以後ゆるやかに回復」という**非単調**な形をしている。
帯域Yはどんな式でも単調増加なので、**山か谷のどちらかが必ずはみ出す**。式の変更で直るのは「序盤に数秒周期を要求される」不合理だけ。
- 残る不可: 第0回(100万クッキーに最短20分超 vs Y(開始付近)=2〜4分)/初回アセント(未踏開拓 B≈1,000〜4,800秒/桁 × ×100に必要な2桁 ≥ 33分〜2.7h)/壁の手前の谷(2〜6分 < 0.5Y)。
- さらに√8x型は係数が大きく、中盤以降の帯域下限(13〜40分)がゲームの自然周回(2〜11分)を上回る。しきい値を上げて周回を伸ばすと桁数が爆発してInfinity化するため(実測: 押しB≈30〜60秒/桁)、**素直には両立しない**。
- 対処方針: (a) 係数を小さくする(Y=120+2.6〜3.5×√x なら中盤が実績のある実現可能域に近い)、(b) 深部の再登坂速度を落とす再調整(コスト膝の強化・ノルマゲージ比の深部増加)で谷を持ち上げ山以後の周回を伸ばす、(c) 第0回・初回アセントは例外として明記(ユーザー了承の「クリック以外」運用)、(d) クリック変更案A+CでS2の遅れを解消。
- 要するに: **式変更は緩和策。谷と山はゲーム経済側の再設計(a+b)とセットでないと帯域には入らない。**

### 5-2. クリックの線形問題と変更案(提案済み・未承認)
- 問題: 設備は上位種ほど基礎生産が桁違いに大きい「階段」を持ち指数的に伸びるが、
  クリック力は「強い指の所持数×固定値+固定倍率」で**所持数に比例(線形)でしか伸びない**。
  深い周回では設備生産に何十桁も置いて行かれ、クリックに使った資金が死ぬ(S2が帯域から外れる根因)。
- 変更案A(推奨): **指先連動** — クリック力 = 従来項 + 毎秒生産 × 0.004 × (1+0.02×√強い指) × (1+クリック系スキル効果)。
  クリックが経済と同じ速度で伸び、強い指を買う意味が全期間残る(⑩⑪にも効く)。
- 変更案C(併用推奨): **神の指を「クリックの上位段」に** — 神の指1個ごとにクリック×1.02(指数)。
- 係数はシミュレーションで①⑩⑪の帯に合わせて調整する。
