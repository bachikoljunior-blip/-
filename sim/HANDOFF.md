- ノルマ基礎を2倍+壁+膝を強めると S1 は周回が伸びて総クッキーが 1e160→1e97 に潰れ(経済崩壊)、未達位置は3%で改善せず。
- **根本原因**: (1)ノルマは経過時間の固定関数だが、各周回のクッキーは同じ時刻で30桁以上ばらつくため1本の時間関数で
  全周回を追えない。(2)層の試練が「絶対最高層」依存のため、深い周回は**再登坂で最高層が周回序盤に跳ね**、試練の重みが
  序盤に立ち上がる=未達が序盤に寄る。→ 固定係数・固定開始層の範囲では原理的に解けない(実測で確認)。

- **新規実装案(推奨)**: 層の試練を「**その周回で前回より深く潜った層**」に効かせる。
  - 変更式: `必要ノルマ = 現行式 × (1+trialCoef)^max(0, 今回最高層 − (前回周回の最高層 + trialStartLayer))`。
  - `前回周回の最高層`(=再登坂の天井)を転生持ち越しの1変数として記録し、その上に trialStartLayer 層ぶんの猶予を置く。
  - 試練は「前回の天井より深く潜った分」だけに効くので、重みの立ち上がりが**新フロンティア開拓(=周回後半)**に来る
    → 未達位置が後半へ移る。速い方針も後半に新層を開けば試練が立ち上がるので T3a=0 が解消する見込み。
  - **追加調整項目**: なし(trialCoef/trialStartLayer を流用)。実装は state に prevMaxStage(前回最高層)を1つ足すだけ。
  - **なぜ解消につながるか**: 未達の位置を「絶対の層数」でなく「再登坂の天井からの超過分」で決めるので、
    周回ごとに深さが違っても未達が常に「新しく開拓する後半」に置ける。時間の固定関数の限界(全周回を1本で追えない)を、
    層基準を各周回の到達点に相対化することで回避する。
- **合格条件変更案(代替・軽い)**: T3aを「全周回で未達が先」から「**転生前に未達が起きる周回が全体の◯割以上**」へ緩める
  (割合はユーザー指定)。新規実装案が通れば不要。まず新規実装案を推奨。

- **実装後の実測=提案8は調整可能範囲でT3aを達成できないと判明(第12次H・要ユーザー判断)**:
  相対化の機構は正しく動作(prevMaxStage が周回持ち越し)。だが係数をどう振っても T3a は解けない。
  - **掃引**(sim/sweep_trial.js, 40h): trialStartLayer 0〜10 × trialCoef 0.08〜**25** の全域で T3a 全体 ~10/140(7%)から
    動かない。しかも trialCoef を上げると未達位置は後半でなく**頭(1〜3%)に寄る**(逆効果)。極端点 coef=25/start=0 でも
    S1=5/47・S10=9/14、いずれも位置1%(=早期周回の基礎ノルマ由来の頭落ち。新層試練は寄与せず)。
  - **根本原因(sim/probe_gap.js で定量化)**: S1の深い周回で、前回天井(178層)を越えるのは**周回のほぼ最後**で、しかも
    **+3層しか新開拓しない**。その時点でクッキーはノルマの**約140桁上**(80%地点で115桁・90%で128桁・100%で141桁)。
    3層の試練でこの差を埋めるには (1+coef)^3 ≈ 1e141 → coef ≈ 1e46 が必要で非現実的。つまり「速い方針も後半に新層を開けば
    試練が立ち上がる」という提案8の見込みが**実測で否定**された(新層開拓が周回末端の+数層しかなく、そこの余裕が桁違い)。
    これは HANDOFF 既述の根本原因(1)「ノルマは時間の固定関数、各周回クッキーは同時刻で30桁以上ばらつく」の具体形。
  - **現在の resting 状態**: 相対化式は残置(ユーザー承認済み・表示は絶対累積のまま)。係数は既定 0.08/10 のまま
    (この設定では試練が発火せず経済に無害。coef を上げると早期周回の spurious な頭落ちを増やすだけなので上げない)。
  - **ユーザー選択(2026-07-07)=(C)到達状態連動の新機構。→ 提案9として実装・成功(下記)。** 提案8の相対化式は残置

#### 提案9. 到達連動ノルマ(T3aを全周回・後半に置く新機構・★2026-07-07 承認・実装済み・第12次H)

**提案8の(C)としてユーザーが選択・実装・実測成功。** 層ゲージ(quotaAtElapsed=時間関数)には一切触れず、
**未達判定にだけ「到達項」を上乗せ**する: `未達 = runCookies < max(従来ノルマ, runCookies×reachCoef×ρ^reachPow)`。
- **ρ(進行比) = 周回内経過秒 ÷ max(前回周回の長さ prevDuration, reachMinSec)**。前回長で正規化=各周回の進行の相対目盛り。
- runCookies が両辺で相殺 → 実質「**ρ が ρ*=(1/reachCoef)^(1/reachPow) を越えたら未達**」。**絶対クッキー桁(周回で140桁ばらつく)に非依存**なのが肝。これが提案8で解けなかった「時間の固定関数では全周回を追えない」を回避する。
- **なぜ層比でなく時間比か(実測)**: 最初 ρ=maxStage/prevMaxStage(層比)で実装したら、未達で層が凍結→prevMaxStage縮小→
  次も早く未達、のスパイラルで最高層が崩壊(S10=3)し未達も頭(10%)に寄った。時間比に変えて解消(prevDuration を持ち越し)。
- **確定係数(第12次H 掃引)**: `reachCoef=20 / reachPow=10 / reachMinSec=600 / reachMaxSec=6000`(ρ*≈0.74)。
  - minSec=1200 だと周回の短い方針(S3)が ρ* に届かず取りこぼす→600 で S3 21→44/47(sweep_reach.js)。
  - **reachMaxSec=6000 = denom(前回周回長)の上限クランプ**(sweep_maxsec.js で T3a と T3b を同時掃引して確定)。
    「長い周回の直後の短い周回」は denom(=直前長)が大きすぎて短い周回内で reach が未発火→T3a取りこぼしになる。上限で抑えると発火する。
    6000で T3a 86→88%・**S10 が 18/23@34% → 25/26@62%(ほぼfull/full・後半)**、かつ **T3b を維持**

### 3-2-4. 「未使用=不合格」の判定ルール(カバレッジ条件を廃止した理由)

**旧⑫(全機能が少なくとも1方針で登場する)は廃止。** ある機能をどの方針も一度も取得しなければ、有効側・無効側とも効果ゼロで同じになり、有効/無効比は必ず **1.0**。1.0 は ①(≥1.2)③(≥1.1)⑨(≥1.05)⑩(≥1.2)の下限を割るので、**登場しない機能は下限で自動的に不合格**になる。よってカバレッジを別条件にするのは冗長。

**ただしハーネスの実装を変える必要がある**: 現在のトグル判定は「その方針が取得しなかった機能」を `(未使用) -` としてスキップしている。これを改め、**各機能につき「少なくとも1方針が取得し、その方針で比≥閾値」を要求。どの方針も取得しなければ比=1.0=不合格として扱う**(スキップしない)。runner.js の `printToggles`/`runToggles` の集計をこの仕様に直すこと。

**例外(下限テストを持たない純QoLスキルノード)**: 開始クッキー(start_1/start_2)・オフライン生産(offline_1)・報酬選択肢+1(reward_choice_2)・オーダーボード(order_board)等、生産に直接効かないノードはトグル下限の対象外。これらの多くは段階式研究の解放ゲート(bake_temperature→オーブン段3、research_analysis→月面段3・反物質段2 等)なので、⑨(段階≥1.05)が解放スキルの取得を間接的に強制する。何もゲートしない純QoL(開始クッキー・オフライン・報酬選択肢増・オーダーボード)は「便利機能」として腐り判定から除外する(生産効率で腐る性質ではなく、取捨は好みの範囲)。
※2026-07-06: 研究解放専用スキル(research_bank 等10ノード)は**ツリーごと削除済み**のため、このリストから消えた(研究段1は対応設備の購入のみで解放)。

### 3-4. 作業ループとプレイ方針の決め方(ユーザー指定・最重要)

プレイヤーは調整後のゲームを見て遊び方を変える。そのため作業は次のループで回す:

```
(1) ゲームの数値・機能を調整する
(2) 調整後のゲームを「プレイヤーとして」見て、10種類のプレイ方針を考え直す(作り直す)
(3) その方針で100時間×10方針のシミュレーションを実行する
(4) 合格条件①〜⑬を判定する
(5) 不合格があれば (1) に戻る(そのたびに (2) の方針見直しも必ずやり直す)
```

**プレイ方針の決め方のルール(第1次からのユーザー指定)**:
- 方針は10種類。プレイヤーのタイプが分かれるように作る(バランス型/クリック特化/金クッキー特化/狩猟特化/研究貯蓄/早回し転生/長期育成/最新設備ラッシュ/ノルマ死守/のんびり放置、といった多様なアーキタイプ)。
- **プレイヤーから見える情報だけ**で設計する: 設備・研究のコストと所持数、現在と次の増分、ノルマ表示、報酬プレビュー、転生PTの見込み表示など。内部式やシミュレーション結果は見ない。
- **シミュレーションの合格条件は一切考慮しない**。方針の変更理由は常に「調整後のゲームをプレイヤーとして見たらこう遊ぶのが得だから」であること。合格条件の達成度から逆算して方針を作るのは禁止。
- 各方針の判断基準は数値で明記する(例: 「研究はコスト≤所持30%なら購入」「獲得見込みPT≥次スキル最安コスト×1.2 かつ 経過120秒以上で転生」)。

- 転生には所持クッキー100万が必要というゲームルールには全方針が従う。
- 乱数要素はすべて期待値で扱う(金クッキーは即時獲得/ブーストを交互、報酬候補は決定的ローテーション、会心は期待値倍率)。

## 4. これまでの調整で分かった重要な技術的知見(再発見に時間を溶かさないこと)

1. **固定倍率は効かない(垂直吸収)**: 周回終盤の成長は毎秒0.3桁(10進)を超える垂直成長になるため、
   生産に定数倍率を掛けても周回時間は数秒しか変わらず、有効/無効比は1.0に張り付く。
   機能を効かせるには「所持数や周回内変数が**成長率**に入る」形(指数の肩、発動系、成長ゲート)が必要。
2. **周回時間のモデル**: 周回時間 D ≈ A(深さ) + B(ラング)×(しきい値 − 自然フロンティア)。
   A=再登坂時間(過去に到達した深さまで戻る時間、深さにほぼ線形 A≈3.1×桁数)。
   B=未踏領域の開拓コスト(秒/桁)。**Bはラング13で約1,000〜1,400秒/桁、ラング25以降は10〜60秒/桁**まで落ちる。
   スキルしきい値を上げても、壁の手前では時間がほぼ動かず、壁を越えると突然何時間もかかる(ラング13でe125を要求したら25時間かかった実測あり)。
3. **×100(④)のノイズ**: 終盤の垂直成長では1秒の判定粒度+金ブースト重なりで周回総クッキーが±1〜1.7桁ぶれる。
   隣接周回比を安全に≥100にするには、しきい値間隔に2.6桁以上の余裕を取り、

#### 帯域の構造問題は式を変えても自動では直らない(重要)
ゲームの周回時間の自然な軌跡は「初回アセントで山(30分〜5時間)→壁の手前で谷(2〜6分)→以後ゆるやかに回復」という**非単調**な形をしている。
帯域Yはどんな式でも単調増加なので、**山か谷のどちらかが必ずはみ出す**。式の変更で直るのは「序盤に数秒周期を要求される」不合理だけ。
- 残る不可: 第0回(100万クッキーに最短20分超 vs Y(開始付近)=2〜4分)/初回アセント(未踏開拓 B≈1,000〜4,800秒/桁 × ×100に必要な2桁 ≥ 33分〜2.7h)/壁の手前の谷(2〜6分 < 0.5Y)。
- さらに√8x型は係数が大きく、中盤以降の帯域下限(13〜40分)がゲームの自然周回(2〜11分)を上回る。しきい値を上げて周回を伸ばすと桁数が爆発してInfinity化するため(実測: 押しB≈30〜60秒/桁)、**素直には両立しない**。
- 対処方針: (a) 係数を小さくする(Y=120+2.6〜3.5×√x なら中盤が実績のある実現可能域に近い)、(b) 深部の再登坂速度を落とす再調整(コスト膝の強化・ノルマゲージ比の深部増加)で谷を持ち上げ山以後の周回を伸ばす、(c) 第0回・初回アセントは例外として明記(ユーザー了承の「クリック以外」運用)、(d) クリック変更案A+CでS2の遅れを解消。
- 要するに: **式変更は緩和策。谷と山はゲーム経済側の再設計(a+b)とセットでないと帯域には入らない。**

### 5-2. クリックの線形問題と変更案(提案済み・未承認)
