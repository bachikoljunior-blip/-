(ゲーム経済のシミュレーション調整)を続けるための指示文。

## ここからをChatGPTに貼る

あなたにはブラウザゲーム「クッキーストラテジャー」の経済バランス調整をしてもらいます。
これまで別のAIセッションで進めてきた作業の続きです。GitHub連携はないので、
ファイルの受け渡しはすべてこのチャット上で行います。以下の指示に厳密に従ってください。

### 0. 最初のターンでやること(この順で)

1. アップロードしたzipを展開し、`sim/HANDOFF.md` を読む。
   まず冒頭の「START HERE」節(最新の引き継ぎ)と **§3-2(合格条件の表)** を読むこと。
   **§3-2 が唯一の判定基準**であり、あなたの仕事はこの表を全部満たすこと。
2. サンドボックスで Node.js が使えるか確認する(Pythonから `subprocess` で `node --version`)。
   結果(自走できるか否か)を私に報告する。
3. `sim/results/` の最新の結果ファイル(ファイル名の R番号が大きいもの)を読み、
   現在どの条件が合格・不合格かを「条件の名前+ひと言説明つき」で要約する。
4. `sim/HANDOFF.md` の「作業キュー」(§3-2 直後の第13次サイクルB節)の先頭から着手する。

### 1. ファイル受け渡しの運用(GitHubなしの回し方・厳守)

- 修正は必ず **「ファイル全体を書き換えた完成品」をダウンロードリンクで返す**こと。
  diffや抜粋コードだけの提示は禁止(私が手で継ぎ接ぎするとミスが混入するため)。
  変更したファイルだけをzipにまとめて返し、変更点の要約(どのファイルのどの値を
  何から何へ・その狙い)を添える。
- 私はそれをローカルのリポジトリに上書きし、ゲームの実挙動を確認して結果を伝える。
- 長い作業では、あなた側の作業状態が消えることがある。**大きな進展があるたびに
  `sim/HANDOFF.md` 自体を更新して返す**こと(何を変えたか・実測値・次にやることを追記)。
  これがセッションをまたぐ唯一の記憶になる。

### 2. シミュレーターの実行方法

Node.js が使える場合は自走する。展開したディレクトリで:

```
cd sim && node runner.js <mode> "" <hours>
```

| mode | 判定する条件 | 標準時間 |
|---|---|---|
| baseline | T1(周回20分〜2h)/T2(解放テンポ)/T3b・T3c/④(1億倍)/新⑥(3分2倍) | 100h(様子見は24h) |
| income | ②改2(方針間の一強禁止)/㉘(稼ぎ口比率) | 100h |
| expect | ①(研究lift)/③(報酬lift)/⑨(段階)/⑬(タイミング)/⑫(文脈依存) | 48h |
| ws | ⑮の2(料理)/㉙(注文ボード)/装備3条件(lift・カバレッジ・毎周回作成) | 24〜36h |
| check19 / check27 / crit23 / affinity | ⑲改/㉗/㉓/㉕㉖(静的チェック) | 数秒 |

- 出力はテキスト。`> results/xxx_R<番号>.txt` に保存し、結果ファイルも私に返すこと。
- 100hシミュは実時間で数十分かかる。**まず24h/48hで方向を確かめ、収束してから
  長時間で確定**するのが確立した進め方。
- **Node.js が使えない場合**: 変更したファイル一式+「私がローカルで実行するコマンド」を
  返すこと。私が実行して結果テキストを貼るので、それを読んで次の調整を返す。
  この往復ループで進める(1往復で複数レバーをまとめて動かしてよい)。

### 3. プロジェクトの構造(どこを触るか)

- `index.html` — ゲーム本体(単一ファイル・ブラウザで動く)。
- `sim/sim.js` — ゲームの忠実なヘッドレス再実装。**ゲームと式・数値を常に一致させる**。
- `sim/params.js` — 調整パラメータ集。**経済調整は基本ここだけを触る**。
- `sim/strategies.js` — 9プレイ方針(S1〜S9)。全方針は「総クッキーを増やす」ことを目指す
  (のんびり放置系の方針は禁止=ユーザー指示)。
- `sim/runner.js` — 合格条件の判定バッテリー(上表)。
- `sim/weave.js` / `sim/tune.js` / `sim/build_prestige_table.js` — 研究配置・スキル梯子・
  転生コスト表の「焼き直し」ツール。経済を大きく動かしたら再実行して
  `weave_costs.json` / `rung_costs.json` を焼き直す(`prestige_costs.json` / `ms_costs.json` は
  最終ビルド時)。
- `sim/results/` — 実測結果の保存場所(R番号=世代)。
- `sim/HANDOFF.md` — 仕様書+合格条件+全歴史。**判定は §3-2 の表だけで行う**。

### 4. ミッション(概観。正確な定義は必ず §3-2 を読む)

100時間×9プレイ方針のシミュレーションで §3-2 の全条件を満たす。主な条件:

- **④ 一億倍成長**: 各周回の総クッキーが前回の1e8倍以上(全方針・全周回ペア)。
- **新⑥ 3分2倍**: 毎秒生産(地力CPS)が180秒ごとのサンプルで毎ペア2倍以上。
- **T1**: 各周回が20分〜2時間。**T2**: 毎周回で新規解放1件以上(第0回は間隔上限のみ)。
- **T3b**: ノルマ維持時間が周回の80%以上(早い方針のみ)。**T3c**: モンスターを全部
  倒せる状態が周回の80%以上。
- **① 研究**: 各研究の期待値liftが取得した全周回で≥1.2。**⑨ 段階**: 段2/3で≥1.05。
- **③ 報酬**: 各報酬liftが取得した全周回で≥1.1+「毎周回その報酬を取る方針」が実在。
- **⑬ タイミング機能**: 最適操作/完全放置の総クッキー比の中央値が[1.05, 2.0]。
- **装備3条件**: 作って装備した周回で+50%以上/486種すべてがどこかで装備される/
  毎周回9カテゴリ全部で1個以上作成。
- **⑮の2 料理**・**㉙ 注文ボード**・**⑫ 文脈依存**・**⑭ 購買力≥1.0**・**②改2/㉘ 一強禁止と
  稼ぎ口シェア**・**⑲改/㉑/㉓/㉕㉖㉗**(静的)。
- **⑤は廃止**。旧⑥⑦⑧㉒・⑩⑪・T3aも廃止(§3-2の表に廃止と明記あり)。

### 5. 作業ループ

計測(runner)→ 結果を読む → `params.js` を調整 → sim/ゲーム同期 → 再計測。
1回の調整で複数レバーをまとめて動かしてよい(ユーザー指示:「まとめて動かして最後に
一括検証でよい」)。行き詰まったら `sim/HANDOFF.md` §4(技術的知見)を読むこと。
同じ発見に時間を溶かさないための地雷マップがある(例:「所持数ループは自走か死かの刃」
「④1e8は毎周回8桁稼ぐ義務=スキル梯子8.5桁/段と一体」「成長の床は量産体制メトロノーム
(45秒ごと×1.25)が作る」)。

### 6. 鉄の規律(ユーザー指定・違反禁止)

1. **合格条件(§3-2)を勝手に変えない**。閾値・定義を変えたくなったら、実測値つきで
   条件を噛み砕いて説明し、具体案を添えて私に相談する(承認するまで判定は旧条件)。
   調整で届かせるのが仕事。承認不要なのは「既存変数の値調整・増加方向の変数追加・
   式変更・ツール類の判断」。
2. **数値・式の変更は `sim/` と `index.html` の両方に反映**する(同期)。simだけ直して
   ゲームを放置しない。逆も同じ。
3. **JSのオブジェクトリテラル行に行中コメントを足さない**(`foo: 1, // コメント` 禁止)。
   コメントは必ず前の行に独立行で書く。過去に機械置換でコードがコメントに飲まれる
   事故が起きた。
4. **数値の丸め規則**(§3-2末尾に記載): 設備/研究/スキルのコストと基礎生産・倍率は
   「有効数字3桁が5の倍数になるよう切り捨て+小数切り捨て」。工房の素材個数は対象外。
   クリックとおばあちゃんの基礎生産は1で固定。
5. 変更したJSは **`node --check` で構文確認**してから返す(Nodeがなければ私に依頼)。
6. 報告では条件を番号だけでなく**名前+ひと言説明**で書く(例:「④(前回比1億倍成長)」)。
7. デザイン(見た目・UI)はユーザーが並行で触っている。あなたは経済数値・ロジックを
   担当し、**UIの見た目を勝手に作り替えない**(数値表示の追随は可)。
8. 私が結果を貼るまで待つ必要がある場面以外は、**質問で止まらず自分の判断で進める**。
   質問してよいのは合格条件の変更承認と、仕様がHANDOFFに無い新規追加だけ。

### 7. 注意点(環境の差)

- 以前のセッションで使っていた補助プローブ(share_probe.js 等)はリポジトリ外にあり
  zipに入っていない。必要なら runner.js を参考に自分で書き直してよい(`sim/` 内に置き、
  返すzipに含めること)。
- 乱数は期待値化されている(シミュは決定的)。同じ入力なら同じ結果になる。
- 長時間シミュがサンドボックスのタイムアウトで切れる場合は、時間を短く刻む
  (例: 24h→12h×方針分割)か、私のローカル実行に切り替える。

### 8. 完了の定義

`runner.js` の baseline(100h)/income(100h)/expect(48h)/ws(36h)/静的チェック4種を
すべて流し、§3-2 の全条件が合格になること。合格したら、最終数値を `index.html` に
同期し、`prestige_costs.json`/`ms_costs.json` を焼き直し、`sim/HANDOFF.md` に最終状態を
記録して、全変更ファイルのzipを返して終了。
